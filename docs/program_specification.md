# 日本語表記変換システム プログラム仕様書 (v1.2)

この文書は、Web ページ上の日本語テキストに対して表記を簡潔にする Chrome 拡張機能のプログラム仕様書である。言語学的情報や実装の詳細、変換規則を体系的に整理し、将来的な改修や拡張に備える。

## 1. 目的と背景

本システムは、既存の日本語テキストを対象に次の目的を達成する。

- **意味を保持したまま表記を圧縮・古層化すること**。送り仮名の省略や古字体の採用により、文章全体の文字数を削減しつつ、文意を損なわない表記を生成する。
- **語彙の漢字化・略語化** により、ひらがな主体の表記を漢語へ変換し、必要に応じてネット略語や頭字語を活用する。
- **単純な文字列置換ではなく、形態素解析に基づく構文変換エンジン** を実装する。文脈に応じた条件付き変換を採用し、誤変換を抑制する。
- **表記の揺れ（異体字や当て字）を仕様として許容・活用** し、乱数シードなどで候補からランダム選択する機能を将来的に導入する。

## 2. 参照情報

本仕様書では以下の外部資料を参考にする。引用文献の行番号は `[リンク†行番号]` の形式で付記する。

* **動詞の活用形とその用法** – 未然形・連用形・終止形・連体形・仮定形・命令形など、動詞活用形の分類と用法を解説している。連用形が連用修飾や連体修飾に用いられ、連体形が名詞を修飾する形であることが指摘されている。
* **日本語動詞の活用形** – 未然形・連用形・終止形・連体形・仮定形・命令形の 6 種類の活用があり、語幹と活用語尾に分けられることが述べられている。
* **尊敬・謙譲の接頭語** – 「御・ご・おん」などを接頭語として名詞に付けると丁寧表現になる例が示される。辞書には「御茶（おちゃ）」が「茶」の美化語であると記述される。
* **送り仮名の付け方・通則6** – 文化庁のガイドラインでは、複合語で送り仮名を省略しても誤読の恐れが無い場合は、省略形を認めるとされる。例えば「書き抜く→書抜く」「申し込む→申込む」など。
* **旧字体の例** – 「乘」は「乗」の旧字体であることが示されている。

## 3. システム全体構成

### 3.1 プロジェクト構造

拡張機能のディレクトリ構成を以下に示す。

```text
jpn-transform-ext/
├─ manifest.json    … 拡張機能のメタ情報と読み込むスクリプト定義
├─ content.js       … コンテンツスクリプト。ルール読込、変換、動的 DOM 監視を担当
├─ transform-bundles.json5 … 変換バンドルの読込順と有効状態を定義
├─ transforms/
│  ├─ 10-surface-normalization.json5 … 表層正規化バンドル
│  ├─ 20-lexical-replacements.json5 … 語彙置換バンドル
│  ├─ 30-okurigana-abbreviation.json5 … 送り仮名省略バンドル
│  ├─ 40-legacy-kanji.json5 … 旧字変換バンドル
│  └─ 50-homophone-kanji.json5 … 同音漢字置換バンドル
├─ lib/
│  ├─ json5.min.js  … JSON5 パーサーライブラリ
│  └─ kuromoji.js   … 日本語形態素解析ライブラリ
└─ dict/            … kuromoji 辞書ファイル一式
```

### 3.2 主要モジュール

| モジュール | 役割 |
|---|---|
| **manifest.json** | 拡張機能名・バージョン・必要な権限・読み込むスクリプトを定義する。`content_scripts` にはページ上で実行する JavaScript を指定し、`web_accessible_resources` には外部読み込みを許可するファイルを列挙する。 |
| **content.js** | ページロード時に実行されるコンテンツスクリプト。変換バンドル定義の読込・形態素解析器初期化・テキストノード変換・動的 DOM 監視を担当する。 |
| **transform-bundles.json5** | 変換バンドルの順序と有効状態を記述する。拡張機能という大きな箱の中で、どの変換箱をどの順序で適用するかを制御する。 |
| **transforms/*.json5** | バンドルごとの変換定義ファイル。`token-rules` と `dictionary-rules` の 2 種類を持てる。旧字変換や同音漢字置換のように固定資産化しやすいものを分離して管理する。 |
| **lib/json5.min.js** | JSON5 をパースするライブラリ。`import` ではなく content script から読み込むため、`manifest.json` で `content_scripts` に含める必要がある。 |
| **lib/kuromoji.js** | 形態素解析ライブラリ。辞書ファイルを用いて日本語文を単語単位に分割し、品詞（`pos`）・細分類（`pos_detail_1`）・活用形（`conjugated_form`）などの情報を提供する。 |
| **dict/** | Kuromoji 辞書ファイル。`kuromoji.builder({ dicPath: "..." }).build(...)` に指定して読み込む。 |

### 3.3 処理フロー

1. **拡張機能の読み込み** – Chrome がページを読み込む際、`manifest.json` の `content_scripts` で指定された順にスクリプト（`json5.min.js` → `kuromoji.js` → `content.js`）が注入される。
2. **バンドル定義読み込み** – `content.js` は `fetch()` で `transform-bundles.json5` を取得し、どの変換箱をどの順序で読むかを決定する。
3. **個別バンドル読み込み** – `content.js` は有効な `transforms/*.json5` を順に読み込み、各バンドル内のルールを優先順位 (`priority`) の降順に整列する。
4. **形態素解析器の初期化** – `kuromoji.builder()` に辞書フォルダを指定してトークナイザを生成する。
5. **初回テキストノード走査** – ページ内の処理対象テキストノードを収集し、`tokenizer.tokenize(text)` で形態素配列に変換する。
6. **変換バンドル適用** – 各トークン配列に対し、バンドル順でルール群を適用する。各バンドル内部では `priority` の降順で評価する。`from` が一致するトークンを見つけ、`conditions` に `prev`（直前トークン）、`current`（当該トークン）、`next`（直後トークン）の条件がある場合はそれも評価する。
7. **文字列の再構築** – トークン配列を変換した後、`surface_form` のみを連結して新しい文字列とし、元のテキストノードに代入する。
8. **動的 DOM 再変換** – `MutationObserver` により後から追加・更新されたテキストノードも監視し、同じ変換パイプラインで再処理する。
9. **ログ出力** – 変換結果や一致した規則、初期化の成否をコンソールに出力する。

### 3.4 安全性と拡張性

本システムでは、`transform-bundles.json5` で順序制御された各バンドル内に **単一トークン置換**、**複数トークン置換（sequence）**、および **単漢字マップによる文字変換** を定義できる。各規則は基本的に以下の形式を取る。

```js
{
  from: "変換前文字列",
  to: "変換後文字列",
  priority: 100,
  conditions: {
    prev: [ { pos: "動詞", cform: "基本形" } ],
    current: [ { pos: "名詞" } ],
    next: [ { pos: "助詞" } ]
  }
}
```

複数トークン一致（`sequence`）では、連続するトークンが順に `surface_form` 等へ一致した場合のみ置換を適用する。複合語や文脈依存語のうち、単純な前後条件より並び自体で管理したほうが安全なものはこの形式で扱う。

`dictionary-rules` では、固定熟語や固定語彙を `phrase_rules` に、単漢字単位の安全な置換を `character_map` に分けて管理する。旧字変換や同音漢字置換は、この形式を基本形とする。

## 4. 変換レイヤの設計

本システムでは変換を以下の 6 段階（Stage1〜Stage6）に分類し、優先度や処理順を明確にする。

| Stage | 内容 | 詳細 |
|---|---|---|
| **Stage1：表層正規化** | 句読点や記号の統一・半角化。例: 全角「、」「。」を半角「､」「｡」へ変換。 | 一部実装済み。 |
| **Stage2：熟語・語彙置換** | 完全一致または最長一致で置換可能な単語。日本語特有の美化語や慣用句、同音異字の復元を含む。 | `transforms/20-lexical-replacements.json5` の `literal` として実装。 |
| **Stage3：接頭語処理** | 敬語の接頭辞（お・ご等）を「御」などに変換する。後続が漢字の場合のみ適用する。 | 条件付き規則で実装予定。 |
| **Stage4：送り仮名省略** | 「変わら→変ら」「書き出す→書出す」のように、活用語尾の一部を省略する。 | 動詞変換の `verb` と `renyou` タイプとして実装。 |
| **Stage5：旧漢字変換** | 新字体から旧字体への変換。例: 「乗→乘」「両→兩」。 | 将来実装予定。 |
| **Stage6：ルビ処理** | ふりがな（漢字《読み》）を HTML の `<ruby>` タグ構造へ変換する。 | 将来実装予定。 |

## 5. 形態素解析と条件判定

### 5.1 形態素情報

Kuromoji により、テキストは次の情報を持つトークン列に分割される。

| フィールド | 説明 |
|---|---|
| `surface_form` | 元のテキスト上の表層形（例: "変換"、"こと"） |
| `basic_form` | 辞書形。動詞の場合は終止形または基本形。 |
| `pos` | 品詞。名詞・動詞・助詞・助動詞・記号など。 |
| `pos_detail_1` | 品詞細分類 1。動詞なら自立・非自立、助詞なら格助詞・係助詞など。 |
| `conjugated_type` | 活用型（五段活用、サ変・スルなど）。 |
| `conjugated_form` | 活用形（未然形・連用形・終止形・連体形・仮定形・命令形）。 |

`conditions` では、`pos`（品詞）、`pos_detail_1`（品詞細分類 1）、`cform`（活用形）、`ctype`（活用型）、`surface`（表層形）、`basic`（辞書形）のいずれかで条件指定できる。省略記法として `pos1 = pos_detail_1`、`cform = conjugated_form` を許容する。

### 5.2 条件評価のロジック

規則適用の際、対象となるトークンの前（`prev`）、当該（`current`）、後（`next`）に対して条件を記述できる。条件は以下の通り処理する。

1. 各 `prev` / `current` / `next` に配列が与えられた場合、その中のどれか 1 つにマッチすれば良い（OR 条件）。
2. 配列内のオブジェクトでは、すべてのフィールドが一致して初めてマッチとみなす（AND 条件）。
3. 指定されていないフィールドは無視する。空の条件は常に成立として扱う。

#### 条件例

* `prev: [ { pos: "動詞", cform: "基本形" } ]` – 直前トークンが動詞の基本形である場合にマッチする。
* `current: [ { pos: "名詞", basic: "こと" } ]` – 当該トークンが名詞で辞書形が「こと」の場合にマッチする。`ことば` は別トークンとして扱われるため誤変換を防げる。
* `next: [ { pos1: "格助詞" } ]` – 直後トークンが品詞細分類 1 で格助詞である場合にマッチする。

### 5.3 複数トークン置換

`sequence` フィールドにより、連続するトークン（例えば「面倒」「ごと」）が順番通り一致した場合のみ置換を適用する。複数トークン置換は、まとまりとして意味が確定している熟語・複合語の管理に使う。また、将来的には `not_conditions` を追加し、特定条件の場合は置換をスキップする負条件判定も検討する。

## 6. 具体的な変換規則例

現行バージョンに含まれる代表的な規則を示す。規則は自由に追加・編集できる。

### 6.1 変換箱の分割方針

変換機能は、拡張機能という大きな箱の中で、以下のような役割別バンドルとして管理する。

| バンドル | 役割 | 性質 |
|---|---|---|
| `surface-normalization` | 記号や句読点の正規化 | 汎用ルール |
| `lexical-replacements` | 指示語や固定語彙の置換 | 汎用ルール |
| `okurigana-abbreviation` | 送り仮名省略 | 汎用ルール |
| `legacy-kanji` | 旧字変換 | `dictionary-rules` による熟語表 + 単漢字表 |
| `homophone-kanji` | 同音漢字置換 | `dictionary-rules` による熟語表 + 単漢字表 |

旧字変換や同音漢字置換は、ルールや対象漢字・熟語の件数が比較的固定されやすいため、他の可変ルールと混ぜず専用バンドルとして分離する。

さらに、この 2 系統は原則として次の単位で扱う。

- 旧字変換: 単漢字単位または固定熟語単位
- 同音漢字置換: 単漢字単位または固定熟語単位

基本は単純置換で済ませ、例外的に誤爆や意味衝突の可能性が高い項目だけを、前後条件や `sequence` を使う文脈依存ルールとして別管理する。つまり、文脈判定は標準手段ではなく例外処理として扱う。

`dictionary-rules` の基本形は次の通りである。

```js
{
  id: "legacy-kanji",
  label: "旧字変換",
  kind: "dictionary-rules",
  phrase_rules: [
    { from: "総合", to: "綜合", priority: 80 }
  ],
  character_map_priority: 10,
  character_map: {
    "両": "兩",
    "乗": "乘",
  }
}
```

### 6.2 語彙置換バンドル（literal）

| 規則 | 変換 | 条件 |
|---|---|---|
| `それ → 其` | 指示詞「それ」を旧字体に変換。 | なし |
| `やっぱり → 矢ッ張` | 副詞を略字漢字表記に変換。 | なし |
| `こと → ヿ` | 名詞「こと」を一文字の記号化した表記に変換。現行ルールでは単独トークンとして解析された「こと」を一律に変換し、「ことば」のような別トークン語は対象外となる。条件を付けて文脈限定する拡張余地は残している。 | なし |
| `面倒 + ごと → 面倒事` | 複合語を sequence として扱い、まとまりで置換する。 | `sequence` |
| `時 + ごと → 時毎` | 複合語を sequence として扱い、まとまりで置換する。 | `sequence` |

### 6.3 送り仮名省略バンドル（verb / renyou）

| 規則 | 変換 | 条件 |
|---|---|---|
| `分かる → 分る` | 動詞「分かる」の送り仮名を省略する。 | なし |
| `当たる → 当る` | 五段活用動詞の送り仮名省略。 | なし |
| `悩み → 悩` | 動詞連用形に由来する名詞の送り仮名を省略する。 | なし |
| `書き出す → 書出す` | 複合動詞の「き」を省く。 | なし |

### 6.4 接頭語変換（literal）

| 規則 | 変換 | 条件 |
|---|---|---|
| `お + 漢字 → 御 + 漢字` | 敬語の接頭辞「お」を旧字「御」に変換する。 | 後続が名詞かつ漢字語であること。 |
| `ご + 漢字 → 御 + 漢字` | 同上。 | 同上。 |

## 7. 拡張ポイントと将来の課題

本システムは MVP（最小実用レベル）として実装されており、拡張余地がある。

* **複数トークン置換の拡張** – より長い複合語や、負条件付きの sequence を扱う機能。
* **旧漢字変換アルゴリズム** – 単漢字表と固定熟語表を整備し、必要最小限の文脈例外ルールを追加する。複数候補がある旧字体の選択方法（擬似乱数やユーザー設定）もここで扱う。
* **同音漢字置換アルゴリズム** – 固定熟語表・固定語彙表を整備し、単純置換で安全に扱えない項目だけを条件付きルールへ分離する。
* **ルビ処理** – 漢字《よみ》形式のタグ化。中間処理では退避し、最終段階で `<ruby>` タグとして復元する必要がある。
* **GUI ベースのルール編集** – 非エンジニアでも `transform-bundles.json5` や `transforms/*.json5` を意識せずにカスタマイズできる Web UI を提供する。

## 8. まとめ

本仕様書では、日本語表記変換システムの目標、構造、処理フロー、各変換段階の設計、形態素解析を用いた条件判定、具体的な規則例、そして拡張ポイントを整理した。本システムは単なる文字列置換ではなく、言語学的背景に基づくエンジンである。将来的な拡張に備え、各ステージの優先順位と設計原則を明示しつつ、現行実装の挙動と文書の整合を維持することを重視する。
