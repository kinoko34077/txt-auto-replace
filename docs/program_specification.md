# 日本語表記変換システム プログラム仕様書 (v1.0)

この文書は、Web ページ上の日本語テキストに対して表記を簡潔にする Chrome 拡張機能のプログラム仕様書である。言語学的情報や実装の詳細、変換規則を体系的に整理し、将来的な改修や拡張に備える。

## 1. 目的と背景

本システムは、既存の日本語テキストを対象に次の目的を達成する。

- **意味を保持したまま表記を圧縮・古層化すること**。送り仮名の省略や古字体の採用により、文章全体の文字数を削減しつつ、文意を損なわない表記を生成する。
- **語彙の漢字化・略語化**により、ひらがな主体の表記を漢語へ変換し、必要に応じてネット略語や頭字語を活用する。
- **単純な文字列置換ではなく、形態素解析に基づく構文変換エンジン**を実装する。文脈に応じた条件付き変換を採用し、誤変換を抑制する。
- **表記の揺れ（異体字や当て字）を仕様として許容・活用**し、乱数シードなどで候補からランダム選択する機能を将来的に導入する。

本仕様書では以下の外部資料を参考にする。引用文献の行番号は `[リンク†行番号]` の形式で付記する。

## 3. システム全体構成

### 3.1 プロジェクト構造

拡張機能のディレクトリ構成を以下に示す。ファイル名は記号的で、内容はこの仕様書の後半で詳述する。

```
jpn-transform-ext/
├─ manifest.json    … 拡張機能のメタ情報と読み込むスクリプト定義
├─ content.js      … コンテンツスクリプト。ページのテキストノードを巡回し変換を実施する
├─ rules.json5     … 変換規則を記述した JSON5 ファイル。コメントや末尾カンマを許容
├─ lib/
│   ├─ json5.min.js  … JSON5 パーサーライブラリ
│   └─ kuromoji.js   … 日本語形態素解析ライブラリ
└─ dict/           … kuromoji 辞書ファイル一式
```

### 3.2 主要モジュール

| モジュール        | 役割 |
|------------------|------|
| **manifest.json** | 拡張機能名・バージョン・アイコン・必要な権限・読み込むスクリプトを定義。`content_scripts` にはページ上で実行する JavaScript を指定し、`web_accessible_resources` には外部読み込みを許可するファイルを列挙する。|
| **content.js** | ページロード時に実行されるコンテンツスクリプト。テキストノードを取得し、形態素解析を通じて `rules.json5` に記述した規則に従って置換を行う。変換前後のログ出力やエラーハンドリングを備える。|
| **rules.json5** | 変換規則を JSON5 で記述する。コメントが使えるため、規則の意図や参考出典をファイル内に記述できる。各規則は `from`・`to`・`priority`・`conditions` (任意)を持ち、条件付き変換や優先順位制御が可能。|
| **lib/json5.min.js** | JSON5 をパースするライブラリ。`import` ではなく content script から読み込むため、`manifest.json` で `content_scripts` に含める必要がある。|
| **lib/kuromoji.js** | 形態素解析ライブラリ。辞書ファイルを用いて日本語文を単語単位に分割し、品詞 (`pos`)・細分類 (`pos_detail_1`)・活用形 (`conjugated_form`) などの情報を提供する。|
| **dict/** | Kuromoji 辞書ファイル。`kuromoji.builder({ dicPath: "..." }).build(...)` に指定して読み込む。|

### 3.3 処理フロー

1. **拡張機能の読み込み** – Chrome がページを読み込む際、`manifest.json` の `content_scripts` で指定された順にスクリプト (`json5.min.js` → `kuromoji.js` → `content.js`) が注入される。これによりパーサーと形態素解析器がグローバル変数として利用可能になる。
2. **ルール読み込み** – `content.js` は `fetch()` で `rules.json5` を取得し、`JSON5.parse()` でオブジェクトとして読み込む。優先順位 (`priority`) の降順に並べる。
3. **形態素解析** – `kuromoji.builder()` に辞書フォルダを指定してトークナイザを生成し、ページ内のテキストノードを1件ずつ取り出して `tokenizer.tokenize(text)` で形態素配列に変換する。各トークンには表層形 (`surface_form`)、品詞 (`pos`)、活用形 (`conjugated_form`) などが含まれる。
4. **変換規則適用** – 各トークン配列に対し `rules` を優先順位順に適用する。`from` が一致するトークンを見つけ、`conditions` に `prev`（直前トークン）、`current`（当該トークン）、`next`（直後トークン）の条件がある場合はそれも評価する。条件に `pos`, `cform`, `basic` などのフィールドが使え、部分一致 (`OR`) 条件も配列で表現できる。
5. **文字列の再構築** – トークン配列を変換した後、`surface_form` のみを連結して新しい文字列とし、元のテキストノードに代入する。
6. **ログ出力** – 変換が行われた場合はコンソールにログを出力し、デバッグしやすいようにする。条件に一致しなかった場合は原文を保持する。

### 3.4 安全性と拡張性

本システムでは、`rules.json5` に定義された **単一トークン置換** を基本とする。複数トークン置換は将来的に拡張されるが、現行バージョンでは未実装である。各規則は基本的に以下の形式を取る。

```js
{
  from: "変換前文字列",        // トークンの surface_form
  to: "変換後文字列",          // 置換後文字列
  priority: 100,               // 優先順位（大きいほど先に適用）
  conditions: {
    prev: [ { pos: "動詞", cform: "基本形" }, ... ],
    current: [ { pos: "名詞" }, ... ],
    next: [ { pos: "助詞" }, ... ]
  }
}
```

複数トークン一致 (`sequence`) が必要な場合は、将来的に `sequence` キーを追加し、連続するトークンが順に `surface_form` 等で一致した場合のみ置換を適用する機能を追加する。現行版ではこれを実装していない。

## 4. 変換レイヤの設計

本システムでは変換を以下の6段階（Stage1〜Stage6）に分類し、優先度や処理順を明確にする。

| Stage | 内容 | 詳細 |
|------|------|------|
| **Stage1：表層正規化** | 句読点や記号の統一・半角化。例：全角「、」「。」を半角「､」「｡」へ変換。 | 将来的に実装予定。 |
| **Stage2：熟語・語彙置換** | 完全一致または最長一致で置換可能な単語。日本語特有の美化語や慣用句、同音異字の復元を含む。例えば「あくまで→飽迄」「ごめん→御免」「決別→訣別」「花弁→花瓣」など【340973998400391†L41-L69】【212879469960127†L66-L68】。 | `rules.json5` の `literal` として実装。 |
| **Stage3：接頭語処理** | 敬語の接頭辞（お・ご等）を「御」などに変換する。後続が漢字の場合のみ適用する。例：「お茶→御茶」「ご飯→御飯」【340973998400391†L41-L69】【769406264250564†L29-L35】。 | 条件付き規則で実装。 |
| **Stage4：送り仮名省略** | 「変わら→変ら」「書き出す→書出す」のように、活用語尾の一部を省略する。文化庁の通則6では複合語で送り仮名を省略できるとされる【431851187530038†L18-L55】。当システムでは、「かな2文字以上は末尾1文字のみ残す」（ルールA）、「かな1文字のみの連用形は末尾省略」（ルールB）、「未然形の『う』は省略しない」（ルールC）を採用する。 | 動詞変換の `verb` と `renyou` タイプとして実装。 |
| **Stage5：旧漢字変換** | 新字体から旧字体への変換。例：「乗→乘」「両→兩」。複数候補がある場合は乱数シードで選択するか固定優先順位で決定する【212879469960127†L66-L68】。 | 将来実装予定。 |
| **Stage6：ルビ処理** | ふりがな（漢字《読み》）を HTML の `<ruby>` タグ構造へ変換する。処理中は退避し、最終出力時に復元する必要がある。 | 将来実装予定。 |

## 5. 形態素解析と条件判定

### 5.1 形態素情報

Kuromoji により、テキストは次の情報を持つトークン列に分割される。主要フィールドのみを示す。

| フィールド | 説明 |
|-----------|------|
| `surface_form` | 元のテキスト上の表層形（例："変換"、"こと"） |
| `basic_form`   | 辞書形（活用語尾を戻した形）。動詞の場合は終止形または基本形【463345566111511†L10-L88】。 |
| `pos`        | 品詞。名詞・動詞・助詞・助動詞・記号など。 |
| `pos_detail_1` | 品詞細分類1。動詞なら自立・非自立、助詞なら格助詞・係助詞など。 |
| `conjugated_type` | 活用型（五段活用、サ変・スルなど）。 |
| `conjugated_form` | 活用形（未然形・連用形・終止形・連体形・仮定形・命令形）【904302289045767†L23-L58】。 |

`conditions` では、`pos`（品詞）、`pos_detail_1`（品詞細分類1）、`cform`（活用形）、`ctype`（活用型）、`surface`（表層形）、`basic`（辞書形）のいずれかで条件指定できる。省略記法として `pos1` = `pos_detail_1`, `cform` = `conjugated_form` を許容する。

### 5.2 条件評価のロジック

規則適用の際、対象となるトークンの前 (`prev`)、当該 (`current`)、後 (`next`) に対して条件を記述できる。条件は以下の通り処理する。

1. 各 `prev`/`current`/`next` に配列が与えられた場合、その中のどれか1つにマッチすれば良い（OR 条件）。
2. 配列内のオブジェクトではすべてのフィールドが一致して初めてマッチとみなす（AND 条件）。
3. 指定されていないフィールドは無視する。空の条件は常に成立として扱う。

#### 条件例

* `prev: [ { pos: "動詞", cform: "基本形" } ]` – 直前トークンが動詞の基本形である場合にマッチ。文法上は連体形でもあるが、Kuromoji では終止形と連体形が同じ "基本形" として出力されることがある【904302289045767†L65-L91】。
* `current: [ { pos: "名詞", basic: "こと" } ]` – 当該トークンが名詞で辞書形が「こと」の場合にマッチ。`ことば` は別トークンとして扱われるため誤変換を防げる。
* `next: [ { pos1: "格助詞" } ]` – 直後トークンが品詞細分類1で格助詞である場合にマッチ。

### 5.3 複数トークン置換への拡張

将来的には `sequence` フィールドを導入し、連続するトークン（例えば「大事」「な」「こと」）が順番通り一致した場合のみ置換を適用する機能を追加する予定である。また、`not_conditions` を追加し、特定条件の場合は置換をスキップする負条件判定も検討する。

## 6. 具体的な変換規則例

現行バージョンに含まれる代表的な規則を示す。規則は自由に追加・編集できる。

### 6.1 名詞変換（literal）

| 規則 | 変換 | 条件 |
|------|------|------|
| `それ → 其` | 指示詞「それ」を旧字体に変換。 | なし |
| `やっぱり → 矢ッ張` | 副詞を略字漢字表記に変換。 | なし |
| `こと → ヿ` | 名詞「こと」を一文字の記号化した表記に変換。【431851187530038†L18-L55】の通則6により複合語で送り仮名を省略できることに着想。`prev` に動詞基本形などの条件を指定して限定する場合もある。 | 任意の条件 |

### 6.2 動詞変換（verb / renyou）

| 規則 | 変換 | 条件 |
|------|------|------|
| `分かる → 分る` | 動詞「分かる」の送り仮名を省略する（ルールA）。 | なし |
| `当たる → 当る` | 五段活用動詞の送り仮名省略。 | なし |
| `悩み → 悩` | 動詞連用形に由来する名詞の送り仮名を省略（ルールB）。 | なし |
| `書き出す → 書出す` | 複合動詞の「き」を省く。文化庁の通則6に基づく省略例【431851187530038†L18-L55】。 | なし |

### 6.3 接頭語変換（literal）

| 規則 | 変換 | 条件 |
|------|------|------|
| `お + 漢字 → 御 + 漢字` | 敬語の接頭辞「お」を旧字「御」に変換【340973998400391†L41-L69】。後続が漢字である場合のみ適用する。 | `current` の前が「お」、`next.pos = 名詞` |
| `ご + 漢字 → 御 + 漢字` | 同上。 | 同上 |

## 7. 拡張ポイントと将来の課題

本システムは MVP（最小実用レベル）として実装されており、拡張余地がある。以下に課題と展望を記載する。

* **複数トークン置換** – 「そのこと」などの二語以上を1単位として置換する機能。現状は単一トークン単位でしか置換できないため、文脈が強い複合語に対応できない。
* **MutationObserver 対応** – ダイナミックに生成されるコンテンツ（SPA）に対し、DOM 変化を監視して再変換する仕組み。
* **旧漢字変換アルゴリズム** – 複数候補がある旧字体の選択方法（擬似乱数やユーザー設定）と、対象の単漢字に対する完全な辞書を整備する。【212879469960127†L66-L68】 によると「乘」は「乗」の旧字体であり、このような対応表が必要。
* **ルビ処理** – 漢字《よみ》形式のタグ化。中間処理では退避し、最終段階で `<ruby>` タグとして復元する必要がある。
* **GUI ベースのルール編集** – 非エンジニアでも `rules.json5` をカスタマイズできるように Web UI を提供する。

## 8. まとめ

本仕様書では、日本語表記変換システムの目標、構造、処理フロー、各変換段階の設計、形態素解析を用いた条件判定、具体的な規則例、そして拡張ポイントを整理した。本システムは単なる文字列置換ではなく、言語学的背景に基づくエンジンである。【904302289045767†L23-L58】 や【463345566111511†L10-L88】 に示される動詞活用形・語幹と語尾の理解を土台とし、文化庁の送り仮名ガイドライン【431851187530038†L18-L55】、尊敬語の接頭辞【340973998400391†L41-L69】【769406264250564†L29-L35】、旧漢字資料【212879469960127†L66-L68】により裏付けられた変換規則を採用している。将来的な拡張に備え、各ステージの優先順位と設計原則を明示し、安定した構造を持つことが本システムの強みである。
